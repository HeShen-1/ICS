# 项目说明

> AI 智能客服系统 — 海慈科技 AI 开发工程师笔试题

## 1. 项目概述

本项目是一个基于大语言模型（LLM）的企业级智能客服系统，采用 RAG（Retrieval-Augmented Generation，检索增强生成）架构。用户上传产品文档、FAQ、政策文件等知识库资料后，系统自动解析并向量化存储；用户提问时，通过语义检索找到相关知识片段，拼接 Prompt 后交由 LLM 生成精准、有来源可追溯的回答。

**核心能力**：文档上传→自动向量化→语义检索→LLM 生成→流式逐字输出→引用来源展示。

## 2. 技术选型及原因

| 层次 | 技术选型 | 选型原因 |
|------|----------|----------|
| **前端** | React 19 + TypeScript + Vite | SPA 生态成熟；SSE 流式消费灵活（fetch + ReadableStream）；Zustand 状态管理轻量够用 |
| **后端框架** | Python FastAPI 0.115 | Python 是 AI/ML 生态事实标准；FastAPI 原生异步 + SSE StreamingResponse，零中间件；Pydantic 自动校验 + OpenAPI 文档自动生成 |
| **LLM** | DeepSeek API (deepseek-v4-flash) | 兼容 OpenAI SDK，零迁移成本；中文理解能力好；性价比极高；由用户提供 API Key |
| **Embedding** | BGE-M3 (BAAI/bge-m3) 本地运行 | 中文检索 MTEB 榜单 SOTA；1024 维向量精度/效率均衡；本地 CPU 推理，零 API 成本，无网络依赖；sentence-transformers 一键加载 |
| **向量数据库** | Milvus Lite（嵌入式模式） | 零部署，数据存本地文件；支持 COSINE 相似度 + metadata 过滤 + 批量删除；pymilvus 2.4 原生 Python 支持；开发 Lite → 生产 Standalone 无缝切换 |
| **关系数据库** | MySQL 8.0 | 成熟稳定；SQLAlchemy ORM 完美支持；存储用户/会话/消息/文档元数据 |
| **文档解析** | LlamaIndex PDFReader (仅 .pdf) + 原生 Python (txt/md) | .txt/.md 原生读取零依赖；.pdf 复用 LlamaIndex Reader（笔试明确允许借用框架） |
| **RAG 链路** | 手动实现（非 LangChain） | 笔试要求"理解每一步逻辑"；手动实现能精确控制分块策略、检索参数、Prompt 模板；避免 LangChain 黑盒调试困难 |
| **UI 组件库** | shadcn/ui（基础组件）+ reactbits.dev（动效） | shadcn/ui 提供 Button/Input/Card/Dialog/Sidebar 等生产级组件，源码可完全控制；reactbits.dev 提供流式文本/消息入场动画 |
| **设计风格** | SaaS Light — Indigo #6366f1 | 灵感来自 Design Prompts #05；白底+Indigo紫强调色+Inter字体+8-14px圆角阶梯；专业客服产品风格 |
| **图标库** | lucide-react | 轻量级开源图标库，Tree-shakable，与 shadcn/ui 完美搭配 |

### 为什么不用 LangChain？

- LangChain 抽象层太厚，预处理/后处理行为不透明，调试困难
- 本项目 RAG 逻辑清晰简单（8 个模块 < 500 行代码），无需引入框架复杂度
- 手动实现更好展示对 RAG 原理的理解（笔试题明确加分项）
- 仅在 .pdf 解析处复用 LlamaIndex Reader（避开发 PDF 解析器写）

## 4. 业务思考：AI 特有问题处理

这是本题重点考查的内容。以下是本系统对三个核心 AI 工程问题的处理策略。

### 4.1 检索为空："诚实"优于"聪明"

**问题**：用户提问内容在知识库中找不到相关信息，LLM 可能"编造"答案（幻觉）。

**处理策略**：
- 向量检索后检查相似度阈值（默认 0.65），低于阈值的结果直接丢弃
- 若所有结果 < 阈值 或 检索结果为空 → **不调用 LLM**
- 直接返回预置兜底话术：
  > 抱歉，我目前的知识库中暂时没有找到与您问题相关的信息。建议您联系人工客服获取帮助，或换一种方式描述您的问题。
- 兜底话术存储在代码中（`fallback.py`），可通过配置修改

**设计理念**：AI 客服和人类客服一样，不知道就说不知道，比胡说更可信。

### 4.2 上下文超长：分层截断策略

**问题**：多轮对话 + 大量检索片段可能导致 Prompt 超过模型上下文窗口（DeepSeek 支持 128K，但最佳性价比在 8-16K），或 Token 消耗过高导致回答质量下降。

**处理策略**：采用分层截断，按优先级逐步裁剪：

```
Token 预算分配（总预算 8000 tokens，可通过 .env 调整）:

  System Prompt          ~500 tokens  ← 固定不裁剪
  历史对话 (最近N轮)      ~1500 tokens ← 远端优先丢弃
  检索知识片段            ~5000 tokens ← 低分优先丢弃
  用户当前问题            ~500 tokens  ← 永远不裁剪
  预留给 LLM 回答         ~500 tokens  ← 保证有空间输出
```

**截断优先级**：检索片段按相似度倒排（低分先丢）→ 历史对话从最早轮次丢弃 → 单片段按句子边界截断 → 超过预算拒绝回答。

**实现方式**：`prompt.py` 中 `build_messages()` 函数在拼接 Prompt 前执行 Token 估算和自动裁剪。

### 4.3 LLM 幻觉：四层防御体系

**问题**：LLM 可能编造知识库中不存在的信息、混淆不同规则、或对模糊问题强行给出确定性回答。

**处理策略**：纵深防御，四层递进：

```
Layer 1: System Prompt 约束（最外层，定基调）
         "你只能根据知识库内容回答。不知道就说不
          知道。不要编造答案。每条回答标注引用来源。"

Layer 2: 检索质量门槛（数据层，断来源）
         相似度 < 0.65 的检索结果直接丢弃。
         不给 LLM 喂不相关内容，从源头减少幻觉。

Layer 3: 来源强制标注（输出层，可追溯）
         每条 AI 回答必须带 📚 参考：[文档名]。
         前端展示引用卡片（文档名+片段+相似度）。
         用户可自行判断回答可信度。

Layer 4: 用户反馈闭环（迭代层，可改进）
         用户踩的回答记录到 feedback 表 + 日志。
         后续分析踩的答案共性，迭代优化 Prompt 模板。
```

### 4.4 System Prompt 设计

```
## 角色
你是智能客服助手，专门为用户解答产品相关问题。

## 核心规则（必须遵守）
1. 你只能根据下方【知识库内容】回答问题
2. 如果知识库中没有相关信息，必须回答：
   "抱歉，我目前的知识库中暂未收录该信息"
3. 每条回答末尾必须标注引用来源：📚 参考：[文档名]

## 回答风格
- 简洁专业，先给出核心结论再展开说明
- 分点说明时使用有序列表
- 涉及流程时先概括再分步

## 知识库内容
{retrieved_chunks_formatted}

## 当前日期
{current_date}
```

### 4.5 加分项：大规模检索下的 LLM 执行保障

当知识库文档非常多（数十条检索片段）时，直接将所有片段塞给 LLM 会导致"注意力稀释"——模型可能遗漏关键规则，或混淆不同文档中的相似规则。

**处理策略**：检索结果分层摘要 + 规则优先级排序

```
Step 1: 检索 Top-K=20 条片段
Step 2: 按相似度排序 → 去重（同一文档的片段合并）
Step 3: Top-5 保持原文，其余 15 条做规则级摘要
Step 4: 以 "核心规则（原文）+ 补充规则（摘要）" 格式给 LLM
Step 5: 关键规则（如退换货时限、金额上限）用 ⚠️ 前缀标注

验证方法: 人工构造 20 条检索结果的场景，对比直接全送 vs 分层摘要的
  - 关键规则遗漏率
  - 幻觉发生率
  - 回答准确度评分
```

### 4.6 加分项：已实现功能

以下加分项已在本项目中完整实现：

| 加分项 | 状态 | 实现方式 |
|--------|------|----------|
| **意图识别** | 已实现 | `rag/intent.py`：基于关键词匹配 + 规则引擎，识别用户问题类型（咨询/投诉/闲聊/未知），不同类型的意图走不同的回答策略。咨询类走 RAG 检索链路，闲聊类直接返回友好兜底，投诉类触发人工客服引导 |
| **追问支持** | 已实现 | `rag/intent.py` 集成追问检测：通过分析对话历史，判断用户当前问题是否为对上一条回答的追问（如"能再详细点吗？""那退换货呢？"）。追问时保留上一轮的检索上下文，提高回答连贯性 |
| **统计面板** | 已实现 | 前端 `StatsPage` + 后端 `services/stats_service.py`：提供管理后台数据看板，包括会话总数、消息总数、用户反馈统计（赞/踩比例）、知识库文档数、每日提问趋势图表。API 路由 `api/stats.py` 提供 RESTful 统计接口 |
| **检索保障** | 已实现 | 多层保障机制：(1) 相似度阈值过滤（<0.65 丢弃）；(2) 检索为空不调 LLM，直接返回兜底话术；(3) LLM 调用 3 次重试 + 指数退避；(4) 回答来源强制标注（📚 参考：[文档名]），前端展示引用卡片，用户可自行验证 |
| **多知识库路由** | 已实现 | `agent/decomposer.py` + `agent/prompt.py`：复杂问题自动拆解为子问题，根据语义路由到对应知识库。支持多 KB 并行检索 + 结果合并，提供更精准的多源回答 |

### 5.2 RAG 链路中 AI 生成代码的优化与修正

在构建 RAG 核心链路时，AI 编程助手生成了初始代码框架，但我进行了以下关键优化和修正：

**1. Chunker 优化**
- AI 初版：简单按固定字符数切分，导致段落被截断
- 修正：改为"段落感知 + 滑动窗口"策略，按双换行分段落，再按 chunk_size 合并/切分，保持语义完整性
- 增加 chunk_overlap 参数，确保上下文连贯

**2. Embedder 优化**
- AI 初版：每次调用都加载模型，导致首次查询延迟 3-5 秒
- 修正：改为懒加载单例模式（`@property`），模型首次加载后常驻内存
- BGE-M3 查询时添加官方推荐的前缀（`encode(..., prompt="...")`）

**3. Prompt 模板优化**
- AI 初版：过于冗长的 System Prompt，包含多余的角色描述
- 修正：简化为 5 条核心规则，去除冗余信息，降低 Token 消耗
- 增加 `{current_date}` 占位符，让 LLM 感知当前时间（涉及退换货时效时会用到）

**4. SSE 流式输出优化**
- AI 初版：每个 token 单独一个 HTTP chunk，网络开销大
- 修正：在 SSE generator 中增加微小延迟（`await asyncio.sleep(0.02)`），使打字机效果更自然，同时减少网络传输次数

**5. Milvus Collection Schema 优化**
- AI 初版：只用向量字段，无 metadata
- 修正：增加 text、source、chunk_index 等动态字段，支持 metadata 过滤和来源追溯

**6. 错误处理增强**
- AI 初版：LLM 调用只 try/except 一次
- 修正：增加 3 次重试 + 指数退避（1s→2s→4s），区分超时/限流/其他错误，返回不同的用户提示

### 5.3 RAG 回答质量评估与验证

**人工测试方法**（笔试题期间采用）：

1. 准备 15 个测试问题，覆盖：
   - 知识库可回答的（如"退换货流程？""产品价格？""如何注册？"）
   - 知识库无法回答的（如"今天天气怎么样？""你们的 CEO 是谁？"）
   - 边界模糊的（如"不满意怎么办？"——需关联退换货+投诉）

2. 对每个回答评估三个指标：
   - **准确性**：回答是否基于知识库内容（而非编造）
   - **完整性**：是否遗漏关键信息（如退换货回答是否包含时间限制）
   - **来源正确性**：引用的文档名和片段是否匹配

3. 记录结果到测试日志，统计准确率

**使用 AI 工具的体会**：

- **最大价值**：AI 编程助手在"代码框架生成"、"重复模式代码"、"文档编写"等机械性工作上效率极高，让我能把时间集中在架构设计和 AI 工程问题上
- **最大陷阱**：AI 容易生成为"看起来正确"但缺少工程细节的代码（如缺少错误处理、硬编码参数、忽略边界条件）
- **核心体会**：**AI 工具是加速器而非替代品**。RAG 链路的 Prompt 设计、幻觉防御策略、截断策略等核心决策，必须有人的判断——AI 只负责执行，不负责思考
- **对本题的影响**：如果没有 AI 工具的帮助，5 天内完成完整的前后端 + RAG 链路 + 全部文档几乎不可能。AI 工具让"想法"到"代码"的转换从小时级降到分钟级
