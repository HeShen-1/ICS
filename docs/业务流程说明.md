# 业务流程说明

> AI 智能客服系统 v1.0 | 核心业务流程

## 1. 问答完整链路

### 1.1 时序图

```mermaid
sequenceDiagram
    actor User as 用户
    participant Web as 前端 (React)
    participant API as FastAPI
    participant VS as Milvus 向量库
    participant BGE as BGE-M3 (本地)
    participant LLM as DeepSeek API
    participant DB as MySQL

    User->>Web: 输入问题 "退换货流程是什么？"
    Web->>API: POST /api/chat/{session_id}

    Note over API: 1. JWT 鉴权
    Note over API: 2. 校验长度 (≤500字)
    API->>DB: 查询今日提问次数
    DB-->>API: count = 5 (< 100, 通过)

    API->>DB: 保存用户消息 (role=user)
    Note over API: 3. 开始 RAG 流程

    API->>BGE: 问题 → Embedding
    BGE-->>API: 1024维向量

    API->>VS: Milvus.search(vec, top_k=5, threshold=0.65)
    VS-->>API: [{text, source, score}, ...]

    alt 检索为空 (score < 0.65)
        API-->>Web: SSE: 兜底话术 (逐字)
        API->>DB: 保存 AI 回答 (含兜底内容)
    else 检索有结果
        Note over API: 4. 拼接 Prompt
        Note over API: System Prompt + 历史(N轮) + 检索片段 + 问题

        API->>LLM: chat(messages, stream=True)
        loop 每个 token
            LLM-->>API: token chunk
            API-->>Web: SSE event: token
            Web->>Web: 追加文本 + 打字机动画
        end

        LLM-->>API: 流结束
        API-->>Web: SSE event: sources (引用来源)
        Web->>Web: 渲染引用卡片

        API-->>Web: SSE event: done (msg_id)
        API->>DB: 保存 AI 回答 (role=assistant, references=JSON)
    end

    User->>Web: 查看引用来源
    User->>Web: 点击 👍/👎
    Web->>API: POST /api/feedback
    API->>DB: 保存反馈记录
```

### 1.2 关键路径耗时分析

```
用户发送问题
  │
  ├─ [50ms]   JWT 验证
  ├─ [20ms]   输入校验
  ├─ [10ms]   查询每日次数 + 保存用户消息
  ├─ [200ms]  BGE-M3 Embedding (CPU, 单句)
  ├─ [30ms]   Milvus 向量检索
  ├─ [10ms]   Prompt 拼接
  ├─ [~2s]    DeepSeek 首 token 延迟
  ├─ [~5s]    DeepSeek 流式输出 (500 token 回答)
  ├─ [10ms]   保存 AI 回答到 MySQL
  │
  ▼
用户看到完整回答：总计约 7-8 秒（主要耗时在 LLM 推理）
流式体验：用户 2 秒内看到首字，持续输出约 5 秒
```

## 2. 文档入库流程

```mermaid
flowchart TD
    A[用户选择文件] --> B{前端校验}
    B -->|格式不符| B1[提示: 仅支持 txt/md/pdf]
    B -->|超过10MB| B2[提示: 文件过大]
    B -->|通过| C[POST /api/knowledge/upload]

    C --> D[后端保存文件到 data/uploads/]
    D --> E[创建 Document 记录 status=processing]

    E --> F{文件类型}
    F -->|.txt/.md| F1[Python open.read]
    F -->|.pdf| F2[LlamaIndex PDFReader]

    F1 --> G[TextChunker.chunk]
    F2 --> G

    G --> G1[按段落分割]
    G1 --> G2[合并短段 + 切分长段]
    G2 --> G3[生成 chunk 列表]

    G3 --> H[Embedder.embed 批量编码]
    H --> I[VectorStore.insert_chunks 写入 Milvus]

    I --> J{写入成功?}
    J -->|是| K[更新 Document: status=ready, chunk_count=N]
    J -->|否| L[更新 Document: status=failed, error_msg=...]
```


## 4. 会话生命周期

```mermaid
stateDiagram-v2
    [*] --> Created: 用户创建新会话
    Created --> Active: 首次发送消息
    Active --> Active: 收发消息
    Active --> Idle: 长时间未活动
    Idle --> Active: 用户返回继续
    Active --> Closed: 用户关闭 / 超时
    Closed --> [*]
```

## 5. 文档处理状态机

```mermaid
stateDiagram-v2

```

## 6. 异常处理流程

```mermaid
flowchart TD
    A[用户发送问题] --> B{LLM 调用}

    B -->|正常| C[SSE 流式输出]
    B -->|超时 >30s| D[重试: 等待 1s]
    D --> D2{第2次}
    D2 -->|正常| C
    D2 -->|超时| E[重试: 等待 2s]
    E --> E2{第3次}
    E2 -->|正常| C
    E2 -->|超时| F[SSE event: error, code=LLM_TIMEOUT]

    B -->|限流 429| G[重试: 等待 4s]
    G --> G2{重试后?}
    G2 -->|正常| C
    G2 -->|仍限流| H[SSE event: error, code=LLM_RATE_LIMITED]

    C --> I[正常结束]

    A --> J{输入校验}
    J -->|问题 >500字| K[400: 问题过长]
    J -->|次数 ≥100| L[429: 次数超限]
    J -->|通过| M{检索阶段}
    M -->|无结果| N[SSE: 兜底话术 + done]
    M -->|有结果| B
```
